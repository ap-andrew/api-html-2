/environments/{environmentName}/stats
Provides metrics gathered on APIs and related entities in Apigee Detail
Apigee collects statistics, or metrics, on API traffic as it flows from apps to APIs. The statistics Apigee collects includes API call information (such as URL, IP address, and user id), total response time, backend response time, maximum response time, error rate, and data exchanged. You can augment the metrics collected by Apigee with custom app- or business-specific data by defining policies that extract, collate, and compute analytics on the custom data that you define. Custom metrics can be collected from message headers, query parameters, as well as the content of XML or JSON request or response messages. The statistics collected are used to generate reports, which can be tailored to your needs by collecting metrics specific to your APIs and the apps that use them. Analytics enables you to view long-term usage trends, segment consumers by top developers and apps, and manage your APIs as a business. /apiproducts
A bundle of API resources (URIs) plus rate limit settings and relevant metadata Detail
API products enable you to bundle and distribute your APIs to multiple developer constituencies simultaneously, without having to modify any code. An APi product consists of a list of API resources (URIs) combined with a Service Plan (rate-limiting policy settings) plus any custom metadata required by the API provider. API products provide the basis for access control in Apigee. As part of the app provisioning workflow, developers select from a list of API product. The developer app is provisioned with a key and secret that enable the app to access the URIs bundled in the selected API product. The API supports multiple API products per app key--meaning that your developers can consume multiple API products without requiring additional keys.
/apis
Enables developers to create and manage APIs in Apigee Enterprise Detail
When you add an API in Apigee, you are creating an API facade, managed by Apigee, that acts as a proxy for the existing API. Instead of calling your existing API, developers instead call the proxied API that runs on APigee. In Apigee, an API is defined by a base URL and a number of API resources (URIs). Resources are the assets offered by your API. For example, if your base URL is example.com/v1/, resources might be /blogs, /projects, /users.
/developers/{developerEmail}/appfamilies
A collection of Apps for a Developer Detail
App families enable you to create functional groups for apps. In many cases, a single app may have multiple manifestations. For example, an app may have sandbox instance and a production, and you may need to associate those instances. You can do this by linking the instances of the app together in a an app family collection. Specify an app family name, associate that name with a developer or a company of your organization, and select one or more app IDs to add to the contents of the app family. You can create app families for developer apps or for company apps within your organization. Prerequisites to use this API call include: The provider has created an organization. You have a valid user ID as an administrator of the organization. The developer exists or the company exists within the organization.
/apps
Applications regsitered with an API Provider's organization Detail
Apps are APi consumer applications registered with an API provider's organization. Apps are registered with an organization to obtain credentials that enable access to one or more API products (or, a set of URIs). The default app profile can be extended by using custom attributes. Apps can be associated with developers (Developer Apps) or with Companies (Company Apps).
/companies
A grouping entity for apps--apps can belong to a Company or to a single Developer Detail
The Company API enables you to associate multiple developers with a single entity, which is in turn managed as part of your organization. While the term Company is used for that entity, it can be any grouping that is appropriate to your organization, e.g., business unit, product line, or division. If the Company API is not used, developers have no association with each other in your organization. The Company API is useful when your goal is to work with multiple developers associated under a single corporate entity for billing purposes, for example. However, it not required that the developers in your organization are associated with a company, so this API is completely optional. Note that a developer is always a single entity, uniquely identified by the username element. Prerequisites to use this API call include: The provider has created an organization. You have a valid user ID as an administrator of the organization.
/companies/{companyName}/appfamilies/
An collection of Apps that belong to a Company Detail
A collection of Apps belonging to a Company. This is often used to associate a 'sandbox' version of an app with a 'production' version of an app. Apigee does no special processing on such collections, except to maintain the relationship among apps that are contained in the family that is created using this method.

/companies/{companyName}/apps/
An App associated with a Company Detail
An application can be associated with a company in the same way that an app can be associated with specific organization developer. An app associated with a company is called a company app. You can use the Company Apps API to work with company apps. For more background information, see Companies for a description of the distinction between organization developers and company developers. Also see Apps for details on the Developer Apps API. Prerequisites to use this API call include: The organization must exist. You have a valid user ID as an administrator of the organization. You have created a company in your organization.
/companies/{companyName}/developers/
Developers grouped into a Company entity Detail
A developer within an organization who is associated with a company. The company developer creates apps on behalf of the company, and any apps created by a company developer are associated with the company and known as 'company apps'. The Company Developer API enables you to add developers to companies. A developer must exist in the system before you can associate them with a company. You associate a developer with a company by adding the developer’s email and assigning the developer a role. Company developer roles are enforced within the client application that you build using the API, not within the Apigee system. Thus, when you retrieve a list of company apps on behalf of a developer, it is your code that checks the developer’s role and displays the capabilities appropriate to that role. You typically create an admin role to allow a company developer to add other developers, delete company apps, and so on. Company developers are mapped to developers in your organization using the developer email as a key. The mapping associates the developer with the company and adds a role. The company developer entity thus has simply an email address and a role while the organization developer entity that it maps to can have a wide variety of attributes defined.

/developers/{developerName}/apps/{appName}/keys/{consumerkey}
The Consumer Key (or "API Key") provisioned for an App Detail
Crenetial pairs consisting of consumer key and consumer secret are provisioned by Apigee to apps for specific API products. APigee maintains the relationship between key and API product, enabling APi product to be added to and removed from consumer keys. A single key can be used to access multiple API products. Keys many be manually or automatically approved for API products--that setting is part of the API product configuration. A key must approved and and approved for an API product to be capable of accessing any of the URIs defined in the API product.
/developers/{developerName}/apps
An App that belongs to a Developer Detail
In Apigee, Apps can belong to individual developers or to groups of developers (known as 'companies'). The Developer App is an App that belongs to a single developer. Apps are created in Apigee in order to obtain credentials to access API Products. So, a Developer App always has associated credentials, API Products, and other metadata (stored as attributes) that can be used for profile-based access control, Analytics, etc. Apps also have status, and may be approved or unapproved. By default Apps are created in the 'approved' state.
/developers
A Developer who is registered with your Organization to create Apps that consume your APIs Detail
Developers can register themselves through a developer portal when they want to access your APIs, but there are cases when you would add a developer via RESTful API. For example, your API could be for use by internal developers and you might not have a developer portal, or you may wish to programmatically add a large number of developers as part of a migration. Developers build apps that access your APIs. Apps are provisioned with keys and secrets, which in turn provide access to the API Products configured for your organization.
/environments
A deployment environment for APIs Detail
By default, Apigee Enterprise Cloud organizations have two environments: 'test' and 'prod'. An environment provides a runtime execution context for APIs. An API Proxy revision must be deployed to an environment before it can be accessed at runtime. No constraints are placed on the usage of test versus prod, for example. Developers are free to implement and enforce and type or testing, promotion, and deployment procedures that suit their development lifecycle.
/maps
A persistence resource that supports storing arbitrary data Detail
Maps provide an API for storing arbitrary name/value pairs that can be access at runtime by custom policies, or for other custom runtime requirements such as protocol support. Each map has one or more 'entries', each of which which defines a name/value pair that can be fetched, updated, deleted via the API.
/oauth/1/accesstokens/{accesstoken}
Oauth 1.0a Access Tokens Detail
Apigee Enterprise validates the API key, retrieves the service profile associated with the API key, and checks to see that the quota has not been exceeded. When you apply OAuth to the API, Enterprise checks the request for an access token. Apigee Enterprise uses access tokens to define a user's permissions for modifying and using a specific API. If an access token is present, and the API is within the scope of the access token, you are allowed to access the API. Prerequisites to use this API call are: The provider has created an organization. You have a valid user ID as a developer. You have created an App. You have a valid consumer key. An access token has been generated.

/oauth/1/verifiers/{verifiercode}
Oauth 1.0a Verifiers Detail
A successful validation/verification of this API call returns the following responses and passes the request to protected resources: The access token and its attributes are made available in flow oauth_token, oauth_token_secret, oauth_response. The consumer token and its attributes are made available in the flow oauth_consumer_key, oauth_consumer_secret. If the request fails the verification, the server sends a response with an appropriate error code and error message. OAuth 1.0 verifier is a verification code tied to the request token. Both the OAuth verifier and request token must be provided in exchange for an access token.
